# Proposal: Infer Schema from Registry

## Problem

KSML currently requires local `.avsc` schema files for its internal type system. When a stream is declared as `valueType: avro:SensorData`, KSML loads `SensorData.avsc` from disk to know the data structure at pipeline construction time.

This is burdensome for users who consume topics where schemas are dynamically generated by external systems (e.g., Kafka Connect producing Avro from database tables). The schema only exists in the schema registry — maintaining a local copy is error-prone and can lead to mismatches between the local file and the registry.

See: https://github.com/Axual/ksml/issues/478

## Solution

Allow KSML to fetch schemas from the schema registry at startup when no explicit schema name is provided in the stream definition.

### User-facing change

Existing syntax continues to work unchanged:

```yaml
streams:
  my_stream:
    topic: my-events
    valueType: avro:SensorData      # loads SensorData.avsc from disk (unchanged)
```

New: omit the schema name to infer from the registry:

```yaml
streams:
  my_stream:
    topic: my-events
    valueType: confluent_avro        # fetches schema from registry using topic name
```

When no schema name is given, KSML resolves the schema by querying the configured schema registry for subjects `{topic}-value` and `{topic}-key`, using the **latest** version. This follows the Confluent `TopicNameStrategy` convention.

## Design

### Deferred schema resolution

Today, `userTypeField` parsing resolves schemas eagerly during YAML parsing — before the topic name is known. The topic name and the type are parsed as independent fields and only assembled together in the `StreamDefinitionParser` lambda.

The change introduces a **deferred resolution** path:

1. When `UserTypeParser` encounters a notation without a schema name (e.g., `confluent_avro`), and the notation supports remote schema fetching, it returns a `UserType` with a marker/sentinel type (e.g., `UnresolvedType`) instead of the notation's default type.

2. In `StreamDefinitionParser`'s assembly lambda — where `topic`, `keyType`, and `valueType` come together — any unresolved types are resolved by calling the notation's `fetchRemoteSchema` method with the appropriate subject name (`{topic}-key` or `{topic}-value`).

```
YAML parsing
    |
    +-- stringField("topic") ---------> "my-events"
    |
    +-- userTypeField("valueType") ---> UserType("confluent_avro", UnresolvedType)
    |
    +-- assembly lambda:
          topic = "my-events"
          valueType has UnresolvedType?
            -> notation.fetchRemoteSchema("my-events-value")
            -> parse into DataSchema -> DataType
            -> resolve UserType
```

### Notation interface extension

Add a default method to the `Notation` interface:

```java
default DataSchema fetchRemoteSchema(String subject) {
    return null; // most notations don't support this
}
```

This keeps existing notations untouched. Only notations backed by a schema registry (Confluent Avro, and potentially Apicurio in the future) override this method using their already-available registry client or configuration.

### Caching

Once fetched, the schema is stored in `SchemaLibrary`'s cache (keyed by subject name) so repeated lookups don't hit the registry.

### Error handling

- If the registry is unreachable or the subject doesn't exist: fail fast at startup with a clear error message.
- If the notation doesn't support `fetchRemoteSchema` (returns `null`): fail with an error indicating that this notation requires an explicit schema name.

## Backward compatibility

Fully backward compatible. The new behavior only activates when a notation is used **without** a schema name in the type declaration. The existing `avro:SensorData` syntax is completely untouched — it continues to load schemas from disk exactly as before.

## Non-goals

- Supporting subject naming strategies other than `TopicNameStrategy` (can be added later).
- Fetching specific schema versions (always uses latest).
- Replacing local schema files for users who prefer them.
