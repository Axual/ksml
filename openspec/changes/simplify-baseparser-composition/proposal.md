## Why

`BaseParser<T>` sits at the root of a 5-hop inheritance chain (`BaseParser → DefinitionParser → TopologyBaseResourceAwareParser → TopologyResourceAwareParser → OperationParser → leaf classes` such as `JoinOperationParser`) in the `ksml` module's parser code (`io.axual.ksml.parser`, `io.axual.ksml.operation.parser`, `io.axual.ksml.definition.parser`). Reading any one leaf class requires reading up to four ancestor files (`DefinitionParser` 376 lines, `TopologyBaseResourceAwareParser` 104 lines, `TopologyResourceAwareParser` 46 lines, `OperationParser` 122 lines) because each ancestor level contributes different helper methods (`functionField`, `topicField`, `storeField`, `structsParser`, ...) that a leaf calls without any indication of which ancestor defines them.

Analysis of the codebase confirmed this depth buys no polymorphism: every consumer that holds a collection of these parsers (e.g. `PipelineOperationParser`, `TypedFunctionDefinitionParser`) stores them as `StructsParser<T>` (an interface), never as `OperationParser`/`FunctionDefinitionParser`/`DefinitionParser` typed values. The inheritance exists purely to hand out protected helper methods, not to express an is-a relationship consumers rely on. A concrete duplication bug was also found: `TopologyBaseResourceAwareParser` and `TopologyResourceAwareParser` each declare their own private `resources` field holding the same object reference, purely to narrow its static type one level down.

This change addresses two independently-scoped, low-risk simplifications identified during that analysis. It intentionally excludes the larger and riskier restructuring of the ~33 `OperationParser` leaf classes and ~21 `FunctionDefinitionParser` leaf classes (tracked as a possible future follow-up), which would need its own proposal.

## What Changes

- Replace the `TopologyBaseResourceAwareParser` → `TopologyResourceAwareParser` inheritance split with two composed helper objects that mirror the real two-phase resource-construction sequence already present in `TopologyResourcesParser` (functions/stores are parsed first into a `TopologyBaseResources`; streams/tables/globalTables are parsed next using only that base set, since they cannot reference topics that don't exist yet; only afterward is a full `TopologyResources`, adding topic lookup, assembled and handed to pipeline/operation parsing). The two composed helpers preserve this same capability boundary: only code that already receives a `TopologyResources` gets a `topicField`-capable helper.
- Eliminate the duplicated `resources` field storage identified above — the object reference is now held once, at composition time, instead of being re-declared and re-assigned one level down purely to narrow its static type.
- Extract the stateless field-builder DSL currently implemented as protected instance methods on `DefinitionParser` (`stringField`, `booleanField`, `integerField`, `longField`, `durationField`, `enumField`, `codeField`, `userTypeField`, `listField`, `mapField`, `customField`, `freeField`, the `structsParser(...)` overloads and their `Constructor0..Constructor10` interfaces, `optional(...)`, `withDefault(...)`, `structSchema(...)`, `validateName(...)`, `parseError(...)`) into a standalone utility class, callable without inheriting from `DefinitionParser`.
- Migrate all call sites currently relying on inherited access to these methods to call the new utility explicitly instead.
- Preserve `DefinitionParser`'s two existing behavioral guarantees — that `parser()` is invoked at most once per instance (memoized) and that `parse()` wraps exceptions via `FatalError.report(...)` — by introducing an equivalent combinator (e.g. a `StructsParser.lazy(Supplier<StructsParser<T>>)` factory, alongside the existing `StructsParser.of(...)`) that callers use instead of relying on inherited `final` methods.
- **BREAKING (internal only)**: `DefinitionParser` subclasses no longer have inherited access to the field-builder methods; any code that subclassed `DefinitionParser` to reach them must call the new utility directly. This is an internal API change with no effect on KSML's YAML syntax, generated JSON Schema, or any other externally observable behavior. One test file (`DefinitionParserFieldValidationTest`) is affected and will be updated as part of this change.

## Capabilities

### New Capabilities
- `ksml-parser-architecture`: the behavior-preservation contract this internal restructuring must uphold — that YAML parsing results and generated schema output are unaffected, and that the compile-time resource-phase safety guarantee (a stream/table/topic definition parser cannot reach topic-lookup capability) survives the move from inheritance to composition. This capability does not describe a new product feature; it captures the regression-safety requirements that make this refactor verifiable.

### Modified Capabilities
(none — no existing spec-level/observable-behavior requirements change. The YAML parsing behavior, the generated JSON Schema output consumed by `ksml --schema` and validated by `ksml-docs-test`, and all public parser APIs such as `TopologyDefinitionParser.parse(...)` remain identical before and after this change.)

## Impact

- **Code**: `ksml/src/main/java/io/axual/ksml/parser/DefinitionParser.java`, `TopologyBaseResourceAwareParser.java`, `TopologyResourceAwareParser.java`, `StructsParser.java`, plus every class currently subclassing `DefinitionParser` and relying on its inherited field-builder methods (main source, ~90 classes across `io.axual.ksml.parser`, `io.axual.ksml.operation.parser`, `io.axual.ksml.definition.parser`, `io.axual.ksml.schema.parser` where applicable) — call sites are mechanically updated to reference the new utility, without changing their own class shape or public API.
- **Tests**: exactly one file requires an edit — `ksml/src/test/java/io/axual/ksml/parser/DefinitionParserFieldValidationTest.java`, whose `FieldTestParser` inner class subclasses `DefinitionParser<Void>` solely to reach `integerField`/`longField` for direct unit testing. All other tests (`DefinitionParserUnresolvedTypeTest`, `AllDefinitionsSchemaValidationTest`, `KafkaProducerRunnerTest`, `BasicStreamRunTest`, `TopologyGeneratorBasicTest`, `KSMLTopologyTestExtension`, `KSMLTestExtension`) only exercise these classes through public, unchanged constructor signatures or the `TopologyDefinitionParser.parse(...)` API and require no changes.
- **Dependencies**: none — no new libraries required.
- **Explicitly out of scope**: converting the ~33 `OperationParser` leaf classes and ~21 `FunctionDefinitionParser` leaf classes from inheritance to composition. That is a larger, higher-risk restructuring left for a possible separate future change.
